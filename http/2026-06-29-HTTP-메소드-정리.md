### Context
HTTP 메소드는 클라이언트가 웹 서버의 자원에 대해 수행하려는 동작을 정의하는 표준화된 방식입니다. 각 메소드는 특정 유형의 요청을 나타내며, 이를 통해 서버는 클라이언트의 의도를 이해하고 적절한 처리를 수행합니다. 이는 웹 서비스에서 자원(Resource)을 효과적으로 관리하고 조작하기 위한 핵심적인 요소입니다.

### Core

HTTP 메소드는 웹 서버의 자원에 대한 요청의 성격을 나타내며, GET, POST, PUT, DELETE가 가장 일반적으로 사용됩니다. 이외에도 PATCH, HEAD, OPTIONS 등의 메소드가 존재합니다.

**주요 HTTP 메소드 및 특징**

1.  **GET**
    *   **목적**: 지정된 자원으로부터 데이터를 취득합니다. (읽기, Read)
    *   **특징**:
        *   **안전(Safe)**: 서버의 상태를 변경하지 않습니다. (단순히 데이터를 조회만 함)
        *   **멱등(Idempotent)**: 같은 요청을 여러 번 보내도 서버의 상태가 동일하게 유지됩니다.
        *   **캐시 가능(Cacheable)**: 응답을 캐싱하여 재사용할 수 있습니다.
        *   **데이터 전송**: URL의 쿼리 스트링을 통해 데이터를 전송합니다.
        *   **보안/프라이버시**: URL에 데이터가 노출되므로, 브라우저 기록에 남거나 서버 로그에 기록될 수 있어 민감한 데이터 전송에 부적합합니다.

2.  **POST**
    *   **목적**: 서버로 데이터를 전송하여 새로운 자원을 생성합니다. (생성, Create)
    *   **특징**:
        *   **안전하지 않음(Not Safe)**: 서버의 상태를 변경합니다 (새로운 자원 생성).
        *   **멱등하지 않음(Not Idempotent)**: 같은 요청을 여러 번 보내면 매번 새로운 자원이 생성될 수 있어 서버의 상태가 계속 변경됩니다.
        *   **데이터 전송**: HTTP Body에 데이터를 담아 전송합니다.
        *   **보안/프라이버시**: URL에 데이터가 노출되지 않아 GET보다 상대적으로 프라이버시 보호에 유리합니다.

3.  **PUT**
    *   **목적**: 지정된 URI의 자원을 갱신하거나, 해당 자원이 존재하지 않으면 새로 생성합니다. (업데이트/생성, Update/Create)
    *   **특징**:
        *   **멱등(Idempotent)**: 같은 요청을 여러 번 보내도 결과적으로는 최종 상태가 동일하게 유지됩니다. (요청 시마다 전체 자원을 덮어쓰므로)
        *   **안전하지 않음(Not Safe)**: 서버의 상태를 변경합니다.
        *   **주요 사용**: 기존 리소스 전체를 업데이트하는 데 이상적입니다.

4.  **DELETE**
    *   **목적**: 지정된 자원을 서버에서 제거합니다. (삭제, Delete)
    *   **특징**:
        *   **멱등(Idempotent)**: 같은 요청을 여러 번 보내도 자원은 한 번만 제거되고 그 이후에는 더 이상 제거될 것이 없어 서버의 상태는 동일하게 유지됩니다.
        *   **안전하지 않음(Not Safe)**: 서버의 상태를 변경합니다.

5.  **PATCH**
    *   **목적**: 자원의 일부를 수정합니다. (부분 업데이트, Partial Update)
    *   **특징**:
        *   `PUT`이 자원 전체를 교체하는 것과 달리, `PATCH`는 자원의 특정 부분만 수정할 때 사용합니다. 이로 인해 `PUT`보다 더 적은 대역폭을 사용할 수 있습니다.
        *   PATCH의 멱등성은 특정 구현 방식(예: JSON Patch)에 따라 달라질 수 있으므로, 일반적으로 멱등하다고 간주하지 않는 경우가 많습니다.

**CRUD와 HTTP 메소드 매핑**

*   **C**reate (생성): `POST`
*   **R**ead (읽기): `GET`
*   **U**pdate (갱신): `PUT` (전체 덮어쓰기), `PATCH` (부분 수정)
*   **D**elete (삭제): `DELETE`

**멱등성(Idempotent)이란?**
동일한 요청을 한 번 보내든, 여러 번 보내든 서버의 상태가 똑같이 유지되는 성질을 말합니다.
*   **멱등한 메소드**: GET, PUT, DELETE
    *   예: `DELETE /posts/1`은 몇 번을 실행해도 결국 1번 글이 삭제된 상태 그대로이므로 멱등합니다.
*   **멱등하지 않은 메소드**: POST, PATCH
    *   예: `POST /posts`는 실행할 때마다 새 글이 계속 생성되므로 멱등하지 않습니다.
    *   예: `PATCH /posts/1`에서 "조회수 1 증가"와 같은 요청은 반복 시 서버 상태를 계속 변경하므로 멱등하지 않습니다.

**안전(Safe) 메소드란?**
서버의 상태를 변경하지 않는 메소드를 말합니다.
*   **안전한 메소드**: GET, HEAD, OPTIONS, TRACE
*   **안전하지 않은 메소드**: POST, PUT, DELETE, PATCH

**기타 HTTP 메소드**

*   **HEAD**: `GET` 메소드와 동일하지만 응답 본문(Body) 없이 헤더 정보만 요청합니다. 리소스가 존재하는지 확인하거나 메타데이터를 얻을 때 사용됩니다.
*   **OPTIONS**: 특정 URL에 대해 어떤 HTTP 메소드들이 사용 가능한지 질의합니다. 주로 CORS(Cross-Origin Resource Sharing) 사전 요청(Preflight Request)에 사용됩니다.
*   **CONNECT / TRACE**: 웹 프록시나 서버 진단 등 특수한 네트워크 통신이나 보안 목적으로 사용되지만, 일반적인 웹 애플리케이션 개발에서는 자주 사용되지 않습니다.

### Insight

HTTP 메소드는 단순히 서버와 통신하는 수단을 넘어, 웹 서비스의 설계 원칙과도 깊이 연관되어 있습니다. 각 메소드의 특성을 이해하고 적절히 활용하는 것은 RESTful API 디자인의 핵심입니다.

*   **데이터 전송 방식의 중요성**: `GET`은 데이터를 URL에 노출시켜 **프라이버시**에 취약하지만 캐싱이 가능하고 북마크에 유리합니다. 반면 `POST`는 데이터를 HTTP Body에 담아 전송하여 상대적으로 **URL 노출로부터 자유로워 프라이버시 보호에 유리**하지만 캐싱이 어렵고 멱등하지 않습니다. 모든 HTTP 통신은 HTTPS를 통해 암호화되어 전송 중 가로채기를 방지할 수 있지만, `GET` 요청의 URL 노출 문제는 여전히 존재합니다. 이 차이는 서비스 설계 시 보안, 성능, 사용 편의성을 고려하는 데 중요한 기준이 됩니다.
*   **멱등성(Idempotency)의 활용**: 멱등성은 클라이언트가 요청 실패 시 안전하게 재시도할 수 있도록 보장하는 중요한 특성입니다. 예를 들어, 결제 요청과 같은 중요한 작업에는 멱등성을 고려하여 중복 처리로 인한 문제를 방지해야 합니다. `PUT`과 `DELETE`는 멱등성을 가지므로, 네트워크 오류 등으로 요청이 여러 번 전송되어도 서버 상태의 일관성을 유지할 수 있습니다.
*   **HTML 폼의 한계**: 순수한 HTML `<form>` 태그는 `GET`과 `POST` 메소드만 지원합니다. `PUT`, `DELETE`, `PATCH`와 같은 메소드를 사용하려면 JavaScript(AJAX, Fetch API 등)를 활용해야 합니다.
*   **메소드 선택의 중요성**: 각 메소드가 가진 의미(의도)와 특성(멱등성, 안전성, 캐싱 가능성)을 정확히 이해하고 상황에 맞는 메소드를 선택하는 것이 중요합니다. 이는 API의 가독성과 예측 가능성을 높이며, 클라이언트와 서버 간의 일관된 상호작용을 보장합니다.
