# BEM: HTML 클래스명 작성 규칙

### Context
웹 애플리케이션의 규모가 커지고 복잡해짐에 따라 CSS 코드의 유지보수, 확장성, 재사용성은 중요한 과제가 되었습니다. 특히 CSS의 전역 스코프(global scope) 특성과 계단식(cascading) 특성으로 인해, 대규모 프로젝트나 여러 개발자가 협업하는 환경에서는 클래스 이름 충돌, 예상치 못한 스타일 오버라이드, 특정 스타일의 강한 의존성 문제 등으로 CSS 관리가 복잡해지고 어려워질 수 있습니다. BEM(Block, Element, Modifier)은 이러한 문제를 해결하고 CSS 코드를 더욱 예측 가능하고 모듈화하기 위한 CSS 방법론 중 하나입니다. "Block", "Element", "Modifier"의 약자로, HTML 클래스명을 명확한 규칙에 따라 작성함으로써 CSS의 구조를 체계적으로 조직하는 데 도움을 줍니다.

### Core
BEM은 모든 UI 요소를 세 가지 유형으로 분류하고 각각에 대한 명명 규칙을 제공합니다.

1.  **Block (블록)**
    *   **정의:** 독립적으로 존재하고 의미를 가지는 UI 컴포넌트(혹은 컨테이너). 페이지의 어느 곳에나 재사용될 수 있으며 다른 블록에 영향을 주지 않아야 합니다.
    *   **예시:** `header`, `menu`, `button`, `card`, `form`, `search-field`
    *   **클래스명 규칙:** `block-name` (하이픈으로 단어 연결)
    *   **HTML 예시:**
        ```html
        <div class="card">
            <!-- ... -->
        </div>

        <button class="button">
            <!-- ... -->
        </button>
        ```

2.  **Element (엘리먼트)**
    *   **정의:** 블록의 일부분으로, 블록 없이는 독립적인 의미를 가질 수 없는 요소입니다. 블록에 종속되며, 블록 내에서만 유효합니다.
    *   **예시:** `card` 블록의 `title`, `image`, `description` 또는 `form` 블록의 `input`, `label`
    *   **클래스명 규칙:** `block-name__element-name` (블록 이름 뒤에 더블 언더스코어 `__`와 엘리먼트 이름)
    *   **HTML 예시:**
        ```html
        <div class="card">
            <img class="card__image" src="..." alt="Card image">
            <h2 class="card__title">Card Title</h2>
            <p class="card__description">This is a card description.</p>
        </div>
        ```
    *   **참고:** 엘리먼트는 블록에 직접 중첩되어야 하며, 엘리먼트 내부에 또 다른 엘리먼트를 중첩하는 방식(`block__element__subelement`)은 권장하지 않습니다.

3.  **Modifier (모디파이어)**
    *   **정의:** 블록 또는 엘리먼트의 모양, 상태, 동작 등을 정의하는 플래그입니다. 동일한 블록이나 엘리먼트가 다른 모습이나 상태를 가질 때 사용됩니다.
    *   **예시:** `button` 블록의 `primary` (기본 색상), `large` (큰 크기), `disabled` (비활성화 상태)
    *   **클래스명 규칙:** `block-name--modifier-name` 또는 `block-name__element-name--modifier-name` (블록 또는 엘리먼트 이름 뒤에 더블 하이픈 `--`과 모디파이어 이름)
    *   **HTML 예시:**
        ```html
        <!-- 블록 모디파이어 -->
        <button class="button button--primary">Primary Button</button>
        <button class="button button--large">Large Button</button>
        <div class="card card--featured">
            <!-- ... -->
        </div>

        <!-- 엘리먼트 모디파이어 -->
        <input class="form__input form__input--error" type="text">
        ```
    *   **주의:** 모디파이어는 항상 기본 블록 또는 엘리먼트 클래스와 함께 사용되어야 합니다. (예: `button--primary` 단독 사용 불가, `button button--primary`로 사용해야 함)

### Insight
BEM 방법론을 도입함으로써 얻을 수 있는 주요 이점과 고려할 점은 다음과 같습니다.

**장점:**

*   **모듈화 및 재사용성:** 각 블록(컴포넌트)이 독립적인 구조를 가지므로, CSS 전역 스코프에서 발생할 수 있는 충돌을 최소화하고 UI 요소를 자유롭게 재사용하고 조합할 수 있습니다.
*   **예측 가능성:** 클래스 이름만 보더라도 해당 요소가 어떤 블록에 속하며 어떤 기능을 하는지, 또는 어떤 상태를 나타내는지 쉽게 예측할 수 있습니다. 이는 새로운 개발자가 프로젝트에 빠르게 적응하는 데 도움을 줍니다.
*   **유지보수 용이성:** 특정 블록이나 엘리먼트의 스타일을 수정할 때, 해당 클래스명으로 인해 영향 범위를 쉽게 파악하고 변경할 수 있습니다.
*   **확장성:** 새로운 컴포넌트나 상태를 추가할 때 기존 코드에 영향을 주지 않고 새로운 규칙을 쉽게 추가할 수 있습니다.
*   **일관성:** 명확한 명명 규칙은 팀 내에서 일관된 CSS 스타일 가이드를 유지하는 데 기여합니다.

**단점:**

*   **클래스 이름 길이:** BEM 규칙에 따라 클래스 이름이 길어질 수 있어 HTML 코드가 다소 장황해 보일 수 있습니다.
*   **학습 곡선:** 처음 접하는 개발자에게는 다소 엄격하고 복잡하게 느껴질 수 있으며, 규칙을 완전히 익히는 데 시간이 필요할 수 있습니다.
*   **엄격한 규칙:** 특정 상황에서는 BEM 규칙이 너무 엄격하여 유연한 스타일링에 제약이 될 수 있다는 의견도 있습니다.

결론적으로 BEM은 CSS의 전역 스코프 문제를 완화하고, 대규모 프로젝트나 여러 개발자가 협업하는 환경에서 CSS 구조를 체계적으로 관리하며, 예측 가능하고 유지보수하기 쉬운 코드를 작성하는 데 매우 효과적인 방법론입니다. 클래스 이름이 길어지는 단점이 있지만, 장기적인 관점에서 코드의 안정성과 효율성을 높이는 데 크게 기여합니다.
